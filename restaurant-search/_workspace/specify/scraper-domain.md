# tabelog 스크래핑 도메인 분석

## 분석 요약

`tabelog.py` 159행 전체를 코드 레벨로 분석하여 5개 영역의 기술적 제약을 정의했습니다.

---

## 1. tabelog URL 체계

### 1.1 프리텍스트 검색 (현재 사용 중)

- **URL**: `https://tabelog.com/{prefecture}/rstLst/?sw={keyword}`
- **실증 근거**: `tabelog.py` 73~79행 -- `BASE_URL/{prefecture}/rstLst/` + `params={"sw": keyword}`
- **현재 prefecture**: `osaka`, `kyoto` (`main.py` PREFECTURE_MAP)
- **검색어 패턴**: `지역명 + 음식 카테고리` (예: `道頓堀 ラーメン`), `지역명 단독` (예: `アメリカ村`), `지역명 + 식사유형` (예: `新梅田食道街 ランチ`)

### 1.2 카테고리코드 URL (미사용, 확장 대상)
- **예상 구조**: `/{prefecture}/rstLst/{category_code}/` (예: `RC0111`)
- 현재 코드에서 미사용. 프리텍스트와 조합 가능 여부 **확인 필요**

### 1.3 지역코드 URL (미사용, 후순위)
- **예상 구조**: `/{prefecture}/{area_code}/rstLst/` (예: `A2701`)
- `next_goals.md` Backlog B1 항목

### 1.4 페이지네이션
- 1페이지: `/{prefecture}/rstLst/?sw={keyword}`
- 2페이지~: `/{prefecture}/rstLst/{page}/?sw={keyword}`
- `MAX_PAGES = 3` (하드코딩, 25행)
- 종료: `.list-rst` 결과 0건 또는 HTTP 에러

### 2.2 고위험 셀렉터

- **`.c-rating__val`** (평점): 코드에 `c-rating-v3__val`(가격용)이 공존하여 rating UI 버전 변경 이력이 확인됨. 가장 깨지기 쉬운 셀렉터.
- **`.c-rating-v3__val`** (가격): 버전 접미사(`v3`) 포함, index 0=디너/1=런치 순서가 하드코딩되어 순서 변경 시 뒤바뀜.
- **`.list-rst__area-genre`**: `/`로 split하므로 복합 카테고리(`焼肉 / ホルモン`) 시 첫 부분만 캡처.

## 4. Rate Limiting / Anti-scraping 제약

### 4.1 현재 설정
- `REQUEST_DELAY = 3.0초` (고정, 23행)
- `MAX_RETRIES = 3` (24행)
- `MAX_PAGES = 3` (25행)
- 재시도: 선형 백오프 `3초 * (attempt + 1)` -- 3/6/9초 (55행)
- 최악 시나리오: 8슬롯 x 4쿼리 x 3페이지 = 96 요청, 약 5분

### 4.2 미구현 대응
- 403 전용 처리 없음
- CAPTCHA 감지 없음
- 연속 실패 시 전체 중단 없음
- 세션/쿠키 관리 없음 (stateless)
- 랜덤 지터 없음 (봇 패턴 인식 위험)

### 4.3 User-Agent
- Chrome 125 고정 (구버전 인식 차단 위험)
- 로테이션 없음, Referer 헤더 없음

## 5. 데이터 품질 제약

### 5.1 누락 가능 필드
- `name_ja`: 셀렉터 2개 모두 실패 시 항목 건너뜀 (폴백 있어 위험 낮음)
- `rating`: 없거나 `"-"` -> 0.0 -> min_score 필터로 제외
- `review_count`: 없으면 0 -> min_reviews 필터로 제외
- `category`: `/`가 없으면 빈 문자열 -> avoid_categories 우회 가능
- `price_lunch`/`price_dinner`: 빈 문자열 빈번 -> CSV에서 `"?"` 출력

### 5.2 비정상 값 패턴
- **가격 순서 반전**: index 0=디너/1=런치 하드코딩 -- **미대응**
- **`name_kr = name_ja`**: 한국어 번역 미구현 (149행) -- tabelog가 한국어를 제공하지 않음
- **`map_query` 정확도**: `name_ja + area_label`인데 area_label이 한국어면 구글맵 검색 부정확
- **이름 기반 중복 제거**: URL이 아닌 가게명으로 판단 -- 동명이인 오탐 가능

### 5.3 구조적 한계
1. 목록 페이지만 파싱 (상세 페이지 주소/영업시간/메뉴 미수집)
2. `_parse_item()` 예외를 `continue`로 무시 (94~95행) -- 에러 원인 추적 불가
3. 인코딩: UTF-8 기반 안정적이나, 전각 특수문자 매칭 이슈 가능성 있음
